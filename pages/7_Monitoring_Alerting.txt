ANNOTATED_ALERTS_FILE = """
notification_channels:
  email_oncall: data-platform-oncall@primeflix.com
  slack_channel: "#data-platform-alerts"
  # Start simple: define where alerts should go before debating the perfect tooling.

alerts:
  - name: daily-orders-pipeline-failed
    source: vertex_pipeline
    severity: critical
    condition:
      pipeline_name: daily-orders
      states: [FAILED, CANCELLED]
      # First alert: if the Vertex pipeline job fails outright, page the team.
    notify: [email_oncall, slack_channel]
    runbook: gs://vertex-pipelines-prod/runbooks/daily_orders_failure.md

  - name: daily-orders-runtime-too-long
    source: vertex_pipeline
    severity: warning
    condition:
      pipeline_name: daily-orders
      duration_minutes_gt: 45
      # Second alert: catch slow runs before they turn into missed SLAs.
    notify: [email_oncall]

  - name: daily-orders-freshness-late
    source: bigquery_check
    severity: critical
    sql: |
      SELECT TIMESTAMP_DIFF(CURRENT_TIMESTAMP(), MAX(updated_at), MINUTE) AS freshness_lag_minutes
      FROM analytics.fct_orders
      # Third alert: check the actual table, not only the pipeline status.
    condition:
      freshness_lag_minutes_gt: 180
    notify: [email_oncall, slack_channel]

  - name: daily-orders-row-count-drop
    source: bigquery_check
    severity: warning
    sql: |
      SELECT COUNT(*) AS row_count
      FROM analytics.fct_orders
      WHERE order_date = CURRENT_DATE() - 1
      # Fourth alert: simple anomaly guardrail for suspiciously low output volume.
    condition:
      row_count_lt: 1000
    notify: [email_oncall]
"""

st.markdown(
    """
**Pipeline infrastructure alerts:** The first two alerts in the example watch the Vertex pipeline job itself. When `source: vertex_pipeline` is set, you're telling your monitoring system to watch Vertex's job status API [the service that reports whether your pipeline run succeeded, failed, or is still running]. The `daily-orders-pipeline-failed` alert fires immediately if the pipeline fails or is cancelled, which means something went structurally wrong like a missing dependency or a code error. The `daily-orders-runtime-too-long` alert is different: it fires when a pipeline run takes longer than expected (45 minutes in this example). This catches performance regressions before they cause downstream business issues like missed data deadlines.

**Data quality alerts:** The last two alerts use `source: bigquery_check`, which means your monitoring system will execute SQL queries against your actual data tables. The `daily-orders-freshness-late` alert checks how old the newest row in your output table is. If the table hasn't been updated in the last three hours (180 minutes), it fires a critical alert because the data stakeholders are not getting fresh information. The `daily-orders-row-count-drop` alert runs a query to count yesterday's rows in your output table and fires if the count is suspiciously low (fewer than 1000 rows). This catches silent data quality issues where the pipeline technically succeeds but produces very little output, which might indicate a filter was applied incorrectly or upstream data was missing.

**Routing and severity:** Each alert has a `severity` field (critical or warning) that helps you prioritize. Critical alerts should page an on-call engineer because they need immediate attention. Warning alerts can wait until business hours. The `notify` field makes the routing explicit instead of hidden in console settings, which means team members and new engineers can see exactly what channels get notified and participate in the decision if thresholds need to change.

**Runbooks as documentation:** The `runbook` field points to a Markdown file that describes how to respond when the alert fires. Instead of having on-call engineers figure out troubleshooting steps from scratch, they have a documented procedure that outlines the most common root causes and recovery steps. Over time, as your team learns what works, the runbook becomes institutional knowledge that survives team turnover.

**Keeping monitoring in version control:** By keeping this alert configuration in your repo alongside your code and pipeline definitions, monitoring thresholds become part of the reviewed, versioned deployment process. If someone wants to change when you alert on freshness, it goes through a pull request, gets reviewed, and is traceable back to a decision and a team member. This beats having thresholds silently lost in a console UI update.
